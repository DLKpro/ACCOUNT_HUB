# Design 05 — Tauri IPC

> Ground-up design #5 and last. Depends on: design 01 (where `gui` lives), design 02
> (keychain), design 03 (vault), design 04 (OAuth flow). This spec is a **facade** over the
> four that precede it — it specifies the Rust ↔ TypeScript contract, not new behaviour.
> Threat model: adversary T2 (malicious webpage / browser-injected JS reaching our IPC).

## Purpose

Define the Rust ↔ React contract: what commands exist, what they return, how errors and
events cross the boundary, how session state is gated, and how TypeScript types stay in
lockstep with Rust types. Map the ~22 existing `apiFetch` call sites in `web/src/api/` to
Tauri `invoke()` commands so the frontend migration in Phase 3 is a mechanical find-replace.

Non-goals:

- Rewriting the React frontend architecture. Existing Zustand / React Query / Vite setup
  stays; only the transport changes from HTTP `fetch` to `invoke`.
- New features. The IPC surface mirrors the existing feature set minus the retired
  auth/email-verify/password-reset paths.
- Mobile or web shells. `core::types` is where DTOs live; a future mobile app would subscribe
  to the same typed contract.

## Type sharing: `specta` + `tauri-specta`

**Decision: specta v2 + tauri-specta.** Generated TypeScript lives at
`web/src/ipc/bindings.ts`, regenerated as part of `cargo build` via `build.rs`. A
`pnpm typecheck` in CI fails if the frontend uses a command that doesn't exist or
passes the wrong shape.

Rust-side derives:

```rust
#[derive(Serialize, Deserialize, specta::Type)]
pub struct LinkedEmail {
    pub id: i64,
    pub email: String,
    pub provider: ProviderId,
    pub is_verified: bool,
    pub created_at: i64,
    // NOTE: no token fields exposed to the frontend — tokens stay in the vault
}

#[derive(Serialize, Deserialize, specta::Type)]
pub enum ProviderId { Google, Microsoft, Apple, Meta }
```

Generated TypeScript (indicative):

```typescript
// web/src/ipc/bindings.ts  — AUTO-GENERATED by build.rs; do not edit
export type LinkedEmail = {
    id: number;
    email: string;
    provider: ProviderId;
    is_verified: boolean;
    created_at: number;
};
export type ProviderId = "Google" | "Microsoft" | "Apple" | "Meta";
```

Escape hatch (captured in ADR 0005): if `specta` / `tauri-specta` stalls, these generated
files are small enough to maintain by hand. The contract survives the tooling.

### Why specta over ts-rs

- Native Tauri integration through `tauri-specta` — auto-registers commands, auto-generates
  `commands.ts` with typed wrappers, not just DTO types.
- Handles our `Result<T, IpcError>` cleanly — the TS wrappers return
  `Promise<T>` that rejects with a typed `IpcError`.
- Events get the same treatment: `app.emit("scan-progress", payload)` in Rust maps to a
  typed listener on the TS side.

`ts-rs` would work for the DTOs but adds hand-written glue for every command and event. Not
worth it when Tauri ships `tauri-specta` integration.

## Command catalog

Replaces every call site in `web/src/api/*.ts`. Names use `snake_case` because Tauri
commands are `snake_case` by convention; `tauri-specta` camelCases them on the TS side.

### Session

| Command | Args | Returns | Lock gate |
|---|---|---|---|
| `session_status` | — | `SessionStatus` | Any |
| `session_unlock` | `{ master_password: string }` | `SessionStatus` | Locked only |
| `session_lock` | — | `SessionStatus` | Unlocked only |
| `session_change_master_password` | `{ old: string, new: string }` | `SessionStatus` | Unlocked only |
| `session_wipe` | `{ confirmation: "WIPE" }` | `()` | Any — irreversible |

```rust
#[derive(Serialize, specta::Type)]
pub struct SessionStatus {
    pub state: SessionState,        // "FirstLaunch" | "Locked" | "Unlocked"
    pub vault_path: String,
    pub last_unlocked_at: Option<i64>,
    pub auto_lock_after_idle_secs: u32,
}
```

Retired, with no replacement: `login`, `register`, `verify-email`, `refresh` (JWT refresh),
`me`, `forgot-password`, `reset-password`, `deleteAccount`. Those concepts don't exist in
the desktop model — `session_unlock` + `session_wipe` cover everything.

### Linked emails

| Command | Args | Returns | Lock gate |
|---|---|---|---|
| `linked_email_list` | — | `Vec<LinkedEmail>` | Unlocked |
| `linked_email_link_start` | `{ provider: ProviderId }` | `LinkHandle` | Unlocked |
| `linked_email_unlink` | `{ id: i64 }` | `()` | Unlocked |

`linked_email_link_start` is non-blocking: it kicks off the OAuth flow, opens the browser,
and returns a `LinkHandle` (contains the `flow_id` and the opened `auth_url`). The actual
linked email lands via a `link-complete` event when the callback completes.

```rust
#[derive(Serialize, specta::Type)]
pub struct LinkHandle {
    pub flow_id: String,
    pub auth_url: String,
    pub loopback_port: u16,
}
```

Rationale: Tauri commands should return within a few seconds; waiting for a user to
complete the browser dance (potentially minutes) would stall the command executor and
bog down other commands. Event-driven completion keeps the IPC channel responsive.

### Scans

| Command | Args | Returns | Lock gate |
|---|---|---|---|
| `scan_start` | `{ linked_email_ids: Vec<i64> }` | `ScanSession` | Unlocked |
| `scan_cancel` | `{ id: i64 }` | `()` | Unlocked |
| `scan_list` | `{ limit: u32 }` | `Vec<ScanSession>` | Unlocked |
| `scan_get` | `{ id: i64 }` | `ScanSessionDetail` | Unlocked |
| `scan_export_csv` | `{ id: i64, path: String }` | `()` | Unlocked; `path` must be within the OS chooser's result |

`scan_export_csv` takes a `path` (rather than returning bytes for the browser to download)
because we're native: the frontend opens the Tauri file-save dialog (via
`@tauri-apps/plugin-dialog`), the user picks a location, then the frontend passes that
path to `scan_export_csv`. Rust validates the path against the allowed-write capability
set before writing. No browser-download semantics.

### Closures

| Command | Args | Returns | Lock gate |
|---|---|---|---|
| `closure_list` | `{ status: Option<ClosureStatus> }` | `Vec<ClosureRequest>` | Unlocked |
| `closure_request` | `{ discovered_account_id: i64, method: ClosureMethod }` | `ClosureRequest` | Unlocked |
| `closure_update_status` | `{ id: i64, status: ClosureStatus }` | `ClosureRequest` | Unlocked |
| `closure_info` | `{ service_name: String }` | `ClosureServiceInfo` | Unlocked |

One-to-one with the existing `web/src/api/closures.ts` surface.

## Event catalog

Events flow Rust → TS only. Subscribed via `listen(name, handler)` on the TS side.

| Event name | Payload | Emitted when |
|---|---|---|
| `session-locked` | `{ reason: "idle" \| "explicit" \| "wipe" }` | Auto-lock, explicit lock, or wipe |
| `scan-progress` | `{ scan_id: i64, emails_scanned: u32, accounts_found: u32 }` | Periodic during a scan (≤ 1/second) |
| `scan-complete` | `{ scan_id: i64, accounts_found: u32 }` | Scan finishes successfully |
| `scan-error` | `{ scan_id: i64, error: IpcError }` | Scan fails |
| `oauth-link-complete` | `{ flow_id: string, linked_email: LinkedEmail }` | OAuth callback succeeds + linked_email row persisted |
| `oauth-link-error` | `{ flow_id: string, error: IpcError }` | OAuth flow fails (user abort, state mismatch, etc.) |
| `token-refresh-revoked` | `{ linked_email_id: i64 }` | Refresh-token 400; UI prompts user to re-link |

Rate limiting (§Security properties): `scan-progress` is throttled to 1 event per second
per scan; the scan task accumulates stats and emits on a timer. Avoids flooding the IPC
channel during high-volume scans.

## Error envelope

All commands return `Result<T, IpcError>`. One enum; intentionally coarse.

```rust
#[derive(Debug, Serialize, thiserror::Error, specta::Type)]
#[serde(tag = "kind", content = "message")]
pub enum IpcError {
    #[error("invalid input: {0}")]
    InvalidInput(String),

    #[error("session is locked")]
    Locked(String),           // session-unlock required

    #[error("not found: {0}")]
    NotFound(String),

    #[error("OAuth provider error: {0}")]
    Provider(String),

    #[error("vault error: {0}")]
    Vault(String),

    #[error("keychain error: {0}")]
    Keychain(String),

    /// Catch-all. The `message` field is a human-readable string safe to show the user;
    /// it NEVER contains secret material, stack traces, or internal types.
    #[error("internal error: {0}")]
    Internal(String),
}
```

TypeScript shape (generated):

```typescript
export type IpcError =
    | { kind: "InvalidInput"; message: string }
    | { kind: "Locked"; message: string }
    | { kind: "NotFound"; message: string }
    | { kind: "Provider"; message: string }
    | { kind: "Vault"; message: string }
    | { kind: "Keychain"; message: string }
    | { kind: "Internal"; message: string };
```

Frontend can discriminate on `kind` for UX-meaningful branching (re-prompt password on
`Locked`, suggest re-link on `Provider`, etc.) without having to parse strings.

### Mapping from core errors

```rust
impl From<core::Error> for IpcError {
    fn from(e: core::Error) -> Self {
        match e {
            core::Error::InvalidInput(m) => Self::InvalidInput(m),
            core::Error::Io(_) => Self::Internal("I/O error".into()),     // path names redacted
            core::Error::NotImplemented(s) => Self::Internal(format!("not implemented: {s}")),
            // when vault / oauth / keychain errors exist:
            // core::Error::Vault(v) => Self::Vault(redact(v.to_string())),
            // etc.
        }
    }
}
```

Key rule: the conversion layer is where **message redaction** happens. Any error variant
that could contain a path, token, or internal type gets scrubbed before it ever reaches
`Serialize`. Enforced by code review — there's no lint that catches "this `String` in a
`Display` impl contains a secret."

## AppState

```rust
pub struct AppState {
    session: Arc<tokio::sync::RwLock<Session>>,
    keychain: Arc<dyn SecretStore>,
    http: reqwest::Client,
    config: Arc<AppConfig>,
}

pub enum Session {
    /// Vault exists; waiting for master-password unlock.
    Locked(LockedSession),
    /// Master password applied; KEK in memory; vault connection open.
    Unlocked(Arc<UnlockedSession>),
    /// No vault yet; first launch.
    FirstLaunch,
}

impl AppState {
    /// Returns the unlocked session or `IpcError::Locked`. The common gate for every
    /// command that touches user data.
    pub fn unlocked(&self) -> Result<Arc<UnlockedSession>, IpcError> {
        let guard = self.session.blocking_read();  // only called from within spawn_blocking
        match &*guard {
            Session::Unlocked(s) => Ok(Arc::clone(s)),
            Session::Locked(_) => Err(IpcError::Locked("unlock the vault to continue".into())),
            Session::FirstLaunch => Err(IpcError::Locked("create a master password to continue".into())),
        }
    }
}
```

Design choices:

- **`tokio::sync::RwLock` on the session** — transitions (unlock, lock, wipe) are rare and
  mutually exclusive with reads; reads dominate. RwLock matches.
- **`Arc<UnlockedSession>`** — multiple commands may be in flight concurrently; each
  clones the Arc so the session survives as long as any command is using it.
- **`keychain` and `http` are session-independent** — they live at app level and don't
  change across lock/unlock transitions.

Lifecycle:

- App launch: `AppState::new()` probes the vault file. If present → `Session::Locked`
  (wait for unlock); if absent → `Session::FirstLaunch` (wait for master-password creation).
- `session_unlock` command: `LockedSession::unlock(master_password)` → `UnlockedSession`;
  swap the enum.
- Auto-lock (idle timer fires): `session_lock` path; swap back to `Locked`; drop the
  Arc<UnlockedSession> so its KEK zeroizes when the last holder drops it.

## Capability model (Tauri 2)

Tauri 2 uses per-window capability files. We ship a single `main` window with a minimal
capability set:

```json
// crates/gui/capabilities/default.json  (indicative)
{
  "identifier": "default",
  "description": "Minimum capabilities for the AccountHub main window",
  "windows": ["main"],
  "permissions": [
    "core:default",
    "core:window:allow-minimize",
    "core:window:allow-close",
    "shell:allow-open",
    "dialog:allow-save",
    "dialog:allow-open",
    { "identifier": "fs:scope-app", "allow": [{"path": "$APPDATA/**"}] },
    "deep-link:default",
    "notification:allow-notify"
  ]
}
```

Explicitly NOT granted:

- `shell:allow-execute` — we never shell out.
- `fs:allow-*` at broad paths — only `$APPDATA/**` (app-specific data dir).
- `http:*` — we don't use Tauri's HTTP plugin; all HTTP goes through our Rust `reqwest::Client`.
- `clipboard:*` — we never touch the clipboard; master password entry uses a secure input
  field (no clipboard suggestion).

### `withGlobalTauri: false`, `freezePrototype: true`

Both set in `tauri.conf.json`:

- `withGlobalTauri: false` — no magic `window.__TAURI__` global; all interaction goes
  through imports from `@tauri-apps/api`. Shrinks the surface a malicious script would find.
- `freezePrototype: true` — `Object.freeze(Object.prototype)` before app code runs.
  Defeats prototype-pollution attacks against any JS the app loads (ours or dep'd).

## Deep links

`accounthub://` URIs arrive via the Tauri deep-link plugin. Dispatcher lives in
`crates/gui/src/deep_link.rs`:

```rust
pub async fn handle_deep_link(app: &AppHandle, url: Url) -> Result<(), DeepLinkError> {
    match url.path() {
        "/unlock" => focus_window_and_show_unlock(app).await,
        "/link" => {
            let provider = url.query_pairs().find_map(|(k, v)| (k == "provider").then_some(v))
                .ok_or_else(|| DeepLinkError::missing_param("provider"))?;
            start_link_flow(app, provider.parse()?).await
        }
        "/verify-email" | "/reset-password" => {
            // Legacy email links — redirect to unlock screen with a "no longer needed" banner.
            redirect_to_unlock_with_legacy_banner(app).await
        }
        _ => Err(DeepLinkError::unknown_path(url.path().to_string())),
    }
}
```

**Deep-link security rule:** every deep-link handler is treated as if the input came from a
malicious webpage. The handler validates parameters strictly (provider must parse as
`ProviderId`; no paths, no filesystem references accepted). The router never dispatches
to internal code paths that mutate state directly — only to UX affordances that the user
must confirm.

## Validation at the IPC boundary

Every command validates its inputs before touching `core`. The webview is untrusted even
though it ships with the app. Checklist embedded in the review process:

- Integer bounds: `limit: u32` must be `1..=1000`; timestamps must be plausible (not before
  2020, not after now + 1 year for `expires_at`).
- Strings: length caps (email ≤ 254 chars, notes ≤ 4096 chars); UTF-8 enforced by serde.
- Enums: use Rust enums (`ProviderId`, `ClosureMethod`, etc.), never `String`. Bad JSON
  fails at deserialization, not at use.
- Paths: for `scan_export_csv`, canonicalize the path and ensure it's within the allowed
  write scope (defined by the Tauri capability); reject otherwise.
- `confirmation` strings for destructive operations: `session_wipe` requires `confirmation: "WIPE"`
  literal — a literal typo catches UI bugs that would otherwise accidentally wipe data.

## Testing strategy

### Unit tests

- Error mapping `core::Error → IpcError`: each variant has a test, verifies no inner type
  leaks through.
- `AppState::unlocked()` returns `Locked` when session is locked, `Unlocked` otherwise.
- Deep-link parser accepts all known paths, rejects all malformed ones.

### Integration tests (`crates/gui/tests/ipc_integration.rs`)

- Use `tauri::test::mock_app()` with a fake `AppState` wiring an `InMemoryVault` + `FakeKeychain`.
- For each command: call with valid args → expected shape; call with invalid args → expected
  `IpcError`; call with session locked → `IpcError::Locked`.
- Event emissions: subscribe with `mock_app.listen(...)`, trigger the command, assert event
  payload shape.

### Contract tests (TypeScript)

- `web/src/ipc/bindings.ts` is generated by `cargo build`. A `scripts/check.sh` step diffs
  the committed file against a fresh generation; any drift fails CI.
- `pnpm typecheck` in CI catches TS consumers that use non-existent commands or wrong arg
  shapes.
- Snapshot tests on the full command-catalog type list so adding a command is a review-
  visible diff.

### Fuzz targets

- `deep_link::parse` — arbitrary URL bytes as input. Never panic; reject unknown paths.
- `scan_export_csv` path parameter — arbitrary strings. Never escape the allowed write
  scope; never panic.

### What CI cannot test

- Tauri's own JS-side event mechanism (requires a running webview). Tested manually during
  release verification.
- Real deep-link dispatch on all three OSes (URI handler registration is per-OS). Same:
  manual release verification on each platform.

## Migration path from the existing `apiFetch`

Phase 3 cutover is a mechanical transformation. Each of the 4 `web/src/api/*.ts` files
becomes:

```typescript
// Before (web/src/api/emails.ts)
export async function list() {
    return apiFetch<LinkedEmail[]>('/emails');
}

// After
import { commands } from '@/ipc/bindings';
export async function list() {
    return commands.linkedEmailList();
}
```

React Query consumers are unchanged — they call the API module; the API module's internals
swap. The ~22 call sites (8 + 2 + 4 + 4 + 4 React Query hooks) stay put.

**Exception:** `scan.ts::exportCsv` currently bypasses the fetch wrapper to trigger a
browser download. The new implementation uses `@tauri-apps/plugin-dialog` for the save
dialog and then invokes `scan_export_csv`:

```typescript
import { save } from '@tauri-apps/plugin-dialog';
import { commands } from '@/ipc/bindings';

export async function exportCsv(scanId: number) {
    const path = await save({
        defaultPath: `scan-${scanId}.csv`,
        filters: [{ name: 'CSV', extensions: ['csv'] }],
    });
    if (!path) return;  // user cancelled
    await commands.scanExportCsv({ id: scanId, path });
}
```

Native file picker; explicit user consent; no browser-download semantics. This is also
the fix for a long-standing annoyance (users couldn't choose the download location
reliably in some browsers).

## Security properties

With the design above, the following hold:

1. **No command executes without the right lock state.** `AppState::unlocked()` is the only
   way to reach session-bound data; commands that require it explicitly gate at the top.
2. **No internal types leak to the frontend.** `IpcError::Internal` carries a human-readable
   message only; the conversion layer (core::Error → IpcError) is the redaction point.
3. **Capability set is minimal.** No shell-exec, no broad filesystem, no HTTP plugin, no
   clipboard. Each entry is a deliberate grant, not a blanket permit.
4. **`withGlobalTauri: false` + `freezePrototype: true`** defeat two categories of JS-side
   attacks (global-object poisoning, prototype pollution).
5. **Every command validates inputs.** Integer bounds, string lengths, enum constraints,
   path canonicalization. Review checklist; automated where possible (Rust enum types for
   provider / status / method).
6. **Deep-link handlers are untrusted-by-default.** A malicious webpage opening an
   `accounthub://` URI cannot trigger destructive operations — all destructive paths go
   through the normal command gate and require the vault to be unlocked first.
7. **Event rate limits** prevent a runaway scan from flooding the IPC channel and DOS-ing
   the UI thread. `scan-progress` is throttled to 1 Hz.
8. **Tokens never cross the IPC boundary.** The frontend never holds a real access token or
   refresh token — only `LinkedEmail` metadata. Provider calls happen in Rust.

What the design does **not** protect against:

- A malicious Rust dep that we've pulled in. Defeated by `cargo-audit` / `cargo-vet` at CI
  time; runtime IPC validation can't stop a library that phones home from inside a command.
- Out-of-process memory inspection of an unlocked app. Acknowledged residual risk in the
  threat model.

## Open questions

### 1. How to handle `session_change_master_password` failure mid-rekey

Design 03 specified the ordering: rekey the vault first, update the keychain second. A
failure between steps leaves a stuck state. The IPC command surfaces this as
`IpcError::Internal("master-password change partially applied; retry")`; the UI offers a
"retry keychain update" action. Detailed flow lands in the Phase 3 PR.

### 2. Concurrent command execution

Two commands can run in parallel (Tauri dispatches on its async runtime). The vault
serializes at the connection mutex; most commands are idempotent. One scenario warrants
thought: `linked_email_unlink(id)` while `scan_start(ids)` is in progress referencing the
same `linked_email_id`. The scan handles a missing linked_email gracefully (moves on with
a warning), but the contract should be documented. Open for Phase 3 refinement.

### 3. Event backpressure

Tauri events are fire-and-forget; a slow frontend listener can't backpressure the emitter.
For `scan-progress` this is fine (throttled to 1 Hz). For future high-rate events (e.g.
per-byte mail-scan progress) we may need a bounded channel + drop-oldest policy. Flag now;
defer decision.

### 4. Type-generation timing

`cargo build` → `specta` → TS file emission: when exactly does this run? Design says
`build.rs`, but `build.rs` runs before `cargo` produces a usable binary; specta needs the
compiled types. The typical pattern is a separate `cargo run --bin export-bindings` step
in CI. Decide in Phase 3 implementation.

### 5. Should the CLI participate in the IPC types?

The CLI doesn't use Tauri IPC, but it uses the same DTOs (via direct `core` calls). The
generated `bindings.ts` is for the GUI only. If a future web-shell or mobile-shell arrives,
we may want a second generation target (e.g. Swift types for mobile). Handle on demand.

## Consequences

**Easier:**
- One place — the `#[tauri::command]` handlers in `crates/gui/` — is the entirety of the
  Rust ↔ TS contract. Adding a command is a ~20-line PR.
- TypeScript types stay in lockstep with Rust types automatically; the frontend can't call
  a command that doesn't exist or pass the wrong shape.
- Error discrimination on the frontend is ergonomic (`switch (e.kind)`), no string-matching.

**Harder:**
- `specta` / `tauri-specta` dep chain adds build-time complexity (codegen step; CI diff
  check). Documented.
- Event rate limits must be enforced at the emitter, not the listener. Reviewer burden:
  "does this event have a max rate?"
- A Tauri upgrade can ripple through the capability config format. Mitigated by pinning the
  Tauri major version and testing upgrades on a branch before merging.

**New risks:**
- Capability grants must be reviewed carefully on every PR that touches them. The
  `deny.toml` / `cargo-deny` path doesn't catch Tauri capability changes; a human reviewer
  does. Add the capability file to the 2-reviewer path in `CODEOWNERS` when it lands.
- Event payload schemas are not version-gated. A breaking payload change is a breaking TS
  API change; the TS CI step catches frontend-side drift, but a dropped field still
  requires a coordinated Rust+TS change. Standard cross-crate versioning discipline.

- Every new command added: reviewer checks input-validation completeness, lock-gate correctness,

## ADR

`docs/adr/0005-tauri-ipc.md` captures the five load-bearing decisions:

1. `specta` + `tauri-specta` for type sharing.
2. Coarse, enum-keyed `IpcError` envelope; redaction at the core → IPC boundary.
3. Long-running ops are command + event pairs, not blocking commands.
4. `AppState` = `RwLock<Session> + Arc<dyn SecretStore> + reqwest::Client`.
5. Minimal capability set; `withGlobalTauri: false`; `freezePrototype: true`.
